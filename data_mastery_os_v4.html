Phase 0 · Foundation ⏱ 20 min · Read First

Data Thinking — Before Any Tool

Every tool in this course — Excel, SQL, Python — is just a way of asking questions about data. Before you pick up any tool, you need to understand what data is, what it can tell you, and how humans turn raw numbers into decisions.

🎯 Learning Objectives

After this section you'll be able to: Distinguish structured from unstructured data · Identify the four types of analytics · Describe the full data lifecycle · Choose the right analysis type for any business question.

What Is Data?

Data is any recorded observation about the world. A temperature reading, a customer purchase, a "like" on a post — all data. The key distinctions you must know:

📋
Structured
Rows and columns. Tables in databases, spreadsheets. Easy to query. Examples: customer records, sales transactions.
🌐
Unstructured
Free-form. Text, images, audio, video. 80% of the world's data. Harder to analyze but often more valuable.
🔢
Quantitative
Numbers you can do math on. Revenue (₹4,200), age (34), temperature (22°C). Can be continuous or discrete.
🏷️
Categorical
Labels, groups, buckets. City (Mumbai), status (pending/delivered), product category. Nominal or ordinal.

The Data-to-Decision Chain

① Raw Data
Numbers, text, events. No meaning on its own. "4200", "Mumbai", "2024-01-05"
② Information
Processed data with context. "Mumbai customer spent ₹4,200 on Jan 5."
③ Knowledge
Patterns across information. "Mumbai customers spend 2× more than average."
④ Decision
Action from knowledge. "Increase marketing budget for Mumbai by 40%."

The Four Types of Analytics

TypeQuestion it answersExampleTools
DescriptiveWhat happened?"Revenue this month was ₹2.4M, down 8%"Excel, SQL
DiagnosticWhy did it happen?"Revenue dropped because returns spiked in Electronics"SQL, Python
PredictiveWhat will happen?"We forecast 12% growth next quarter"Python + ML
PrescriptiveWhat should we do?"Discount Electronics 15% to recover volume"ML + Business
💡 Where Most Companies Are

Most companies are still stuck on descriptive analytics — dashboards that show what happened last month. The jump to diagnostic requires SQL + pattern thinking. Predictive needs Python + statistics. Prescriptive is the frontier — it requires everything plus business judgment. This course gets you through predictive.

The Data Lifecycle — Know Your Stage

Every data project follows this arc. Knowing which stage you're in determines which tool to use:

📡
1. Collect
Surveys, sensors, transactions, scraping. Data doesn't exist until something is recorded.
🗄️
2. Store
Databases (SQL), data warehouses (BigQuery, Snowflake). Structure determines what questions you can later ask.
🧹
3. Clean
Fix nulls, duplicates, wrong types, outliers. Real data is always messy. This takes 70% of your time.
🔍
4. Analyze
EDA → model → validate. The "fun part" that only happens after cleaning.
📊
5. Visualize
Charts, dashboards, reports. Transforms numbers into decisions by making patterns visible.
🚀
6. Deploy
APIs, dashboards, automated reports. The insight reaches the person who needs to act on it.
⚠️ The Biggest Beginner Mistake

Jumping straight to modeling. Data scientists who build great models on dirty data get wrong answers with high confidence — the most dangerous outcome in analytics. Clean first. Always.

Phase 1 · Excel Lesson 01

Why Excel First

"Why learn Excel if I want to be a data scientist?" Because Excel taught a billion people to think in tables. If you can't think in rows and columns, SQL and Pandas will always feel foreign.

🎯 Learning Objectives · ⏱ 15 min

After this lesson you'll be able to: Explain why Excel is the right starting point · Map Excel concepts (cell, sheet, formula) to Python and SQL equivalents · Recognize when Excel is the right tool vs. when to upgrade.

What Excel Actually Is

Excel is a grid database with a formula language built in. Every cell is a variable. Every formula is a function. Every sheet is a table. When you understand this, Excel stops being a "spreadsheet app" and starts being a visual data engine.

🔢
Cell = Variable
A1 holds a value. You reference it by name, just like a variable in Python.
📋
Sheet = Table
Each sheet is a structured table. Rows are records. Columns are fields. Same as SQL.
⚙️
Formula = Function
=SUM(A1:A10) is literally calling a function with a range as input. That's programming.
🔄
Drag = Loop
Dragging a formula down a column is the Excel equivalent of a for loop. Same idea.

Phase 1 · Excel Lesson 02

The Grid Mind

Before formulas, master the structure. The difference between a messy spreadsheet and a powerful one is the person's understanding of what data actually is.

Tidy Data: The One Rule

A tidy dataset has: one row per observation, one column per variable, one table per type of data. This rule alone makes 80% of your analysis easier.

⚠️ The Most Common Mistake

Storing data horizontally (months as columns) instead of vertically (Month as a column with values). This breaks every formula, every pivot, every chart. Always go vertical.

❌ Bad Structure✅ Good Structure (Tidy)
Name | Jan | Feb | MarName | Month | Revenue
Alice | 100 | 120 | 90Alice | Jan | 100
Bob | 80 | 95 | 110Alice | Feb | 120

Absolute vs Relative References

This is the single most important concept in Excel. When you copy a formula, Excel moves the references relative to where you paste. Lock them with $ when you don't want that.

Excel Formulas
=A1          // Relative: moves when copied
=$A$1        // Absolute: locked, never moves
=$A1         // Mixed: column locked, row moves
=A$1         // Mixed: row locked, column moves
// Calculating % of total — $B$12 is the total cell
=B2/$B$12
Phase 1 · Excel Lesson 03

Power Formulas

Most people know SUM and AVERAGE. The people who get hired know XLOOKUP, INDEX-MATCH, SUMIFS, and array formulas. Here's the full arsenal.

The Lookup Family

🔍 The Lookup Analogy

VLOOKUP is like searching a phone book: you find the name, then read across. XLOOKUP is like asking a smart assistant — you describe what you want and it finds it in any direction.

Excel
// VLOOKUP — the old way (still everywhere)
=VLOOKUP(A2, Products!A:C, 3, FALSE)
// XLOOKUP — the modern way
=XLOOKUP(A2, Products!A:A, Products!C:C, "Not found")
// INDEX-MATCH — the power user combo
=INDEX(C:C, MATCH(A2, A:A, 0))
Conditional Aggregation
// SUMIFS — multiple conditions (the real one)
=SUMIFS(C:C, B:B, "North", D:D, "2024")
// COUNTIFS — count with multiple conditions
=COUNTIFS(B:B, "North", C:C, ">"&1000)
💡 The SQL Connection

SUMIFS is exactly SELECT SUM(C) FROM table WHERE B='North' AND D='2024'. Once you see this, moving between Excel and SQL becomes natural.

Phase 1 · Excel Lesson 04

Pivot Tables — Your First Aggregation Engine

A Pivot Table is a drag-and-drop GROUP BY. When you master it, you'll understand aggregation so deeply that SQL's GROUP BY will feel like you already know it.

🔄 The Pivot Mental Model

Rows: what you're grouping by. Columns: a secondary grouping. Values: what you're aggregating (SUM, COUNT, AVG). Filters: what to exclude. This is exactly GROUP BY + HAVING in SQL.

Phase 2 · SQL Lesson 05

Declarative Thinking

SQL is the only major language where you describe the what, not the how. This single idea is the key to thinking in SQL. Stop thinking about steps. Think about the shape of your result.

📖 The Librarian Analogy

You don't tell a librarian which shelf to walk to, which index to look in, which direction to turn. You say "I want all crime novels from 2020 with over 4 stars." The database optimizer is the librarian. Your job: describe the result.

SQL vs Excel vs Python — Three Lenses

📊 Excel
Visual, cell-by-cell. Great for exploration, dashboards, quick answers. Works best under ~100k rows.
🗄️ SQL
Set-based, declarative. Great for querying millions of rows in databases. Where 90% of business data lives.
🐍 Python
Procedural + vectorized. Great for complex transformations, ML, automation. Works anywhere.

Phase 2 · SQL Lesson 06

DDL & Schema Design

Data Definition Language is how you blueprint a database. The schema decisions you make upfront determine how fast and clean your queries will be forever after. Think like an architect, not a plumber.

SQL — DDL
CREATE TABLE customers (
    customer_id   INTEGER      PRIMARY KEY,
    name          TEXT         NOT NULL,
    email         TEXT         UNIQUE NOT NULL,
    city          TEXT,
    signup_date   DATE         DEFAULT CURRENT_DATE
);
CREATE TABLE orders (
    order_id      INTEGER      PRIMARY KEY,
    customer_id   INTEGER      REFERENCES customers(customer_id),
    total_amount  DECIMAL(10,2) NOT NULL,
    order_date    DATE         DEFAULT CURRENT_DATE
);
⚠️ Never Store Money as FLOAT

Floating point arithmetic is imprecise. 0.1 + 0.2 = 0.30000000000000004. Always use DECIMAL(10,2) or store prices in cents as INTEGER.

Phase 2 · SQL Lesson 07

SELECT Mastery

SELECT is deceptively simple. Every complex query you'll ever write is built from the same six clauses. Learn them in execution order, not written order, and everything clicks.

StepClauseWhat it doesExcel equivalent
1FROMChoose the table(s) to work withSelect your sheet
2WHEREFilter rows before aggregatingFilter / SUMIFS criteria
3GROUP BYCollapse rows into groupsPivot Table Rows
4HAVINGFilter groups after aggregatingPivot Table filter on values
5SELECTChoose which columns to returnChoose which Pivot columns
6ORDER BYSort the final resultSort in Pivot/filter
💡 The Key Insight

WHERE filters rows before grouping. HAVING filters groups after aggregating. This is why you can't use WHERE revenue > 1000 when revenue is a SUM — you need HAVING.

SQL
-- The full SELECT template
SELECT city, COUNT(*) AS total_orders, SUM(total_amount) AS revenue
FROM customers c JOIN orders o ON c.customer_id = o.customer_id
WHERE o.order_date >= '2024-01-01'
GROUP BY city
HAVING SUM(total_amount) > 500
ORDER BY revenue DESC
LIMIT 10;
Phase 2 · SQL Lesson 08

JOINs — The Art of Combining Tables

Normalized databases split data across tables. JOINs are how you put it back together. Most real-world SQL queries you'll write professionally involve at least two JOINs.

INNER JOIN
Matches only
LEFT JOIN
All left + matches
FULL OUTER
Everything
SELF JOIN
Table joins itself

INNER JOIN — The Most Common

Only returns rows where the join condition matches in both tables. Use this when you only want records that exist on both sides.

SQL
SELECT c.name, o.order_id, o.total_amount
FROM customers c
INNER JOIN orders o ON c.customer_id = o.customer_id;
Phase 2 · SQL Lesson 09

Aggregation & CTEs

CTEs (Common Table Expressions) are the most important readability tool in SQL. They let you break a complex query into named steps — like functions in code.

🧩 The Lego Analogy

A CTE is like naming a Lego piece before using it. Instead of "that blue 4x2 brick with the flat top," you say "the base." Then you build with named pieces. Your query becomes a readable story.

SQL — CTEs
-- Step 1: Find high-value customers
WITH high_spenders AS (
    SELECT customer_id, SUM(total_amount) AS lifetime_value, COUNT(*) AS order_count
    FROM orders
    GROUP BY customer_id
    HAVING SUM(total_amount) > 1000
),
-- Step 2: Get their details
enriched AS (
    SELECT c.name, c.city, hs.lifetime_value, hs.order_count
    FROM customers c JOIN high_spenders hs USING(customer_id)
)
-- Step 3: Final output
SELECT * FROM enriched ORDER BY lifetime_value DESC;
Phase 2 · SQL Lesson 10

Window Functions — The Pro's Tool

Window functions let you do aggregations without collapsing rows. They see the context around each row. Once you learn them, you'll use them in 80% of your serious queries.

🎥 The Camera Analogy

A window function is a camera that moves through the dataset frame by frame. For each row, it can see both the current row AND nearby rows. GROUP BY collapses everything into one frame. Window functions keep every frame.

SQL — Window Functions
-- Rank customers by revenue within each city
SELECT c.name, c.city, SUM(o.total_amount) AS revenue,
    RANK() OVER (PARTITION BY c.city ORDER BY SUM(o.total_amount) DESC) AS city_rank,
    SUM(SUM(o.total_amount)) OVER (PARTITION BY c.city) AS city_total,
    LAG(SUM(o.total_amount)) OVER (ORDER BY o.order_date) AS prev_month
FROM customers c JOIN orders o USING(customer_id)
GROUP BY c.customer_id, c.name, c.city, o.order_date;
Phase 2 · SQL Lesson 11

Subqueries & Indexes

Subqueries let you use a query's result as a table or a value inside another query. Indexes make that query run in milliseconds instead of minutes. Together, they're the tools that separate weekend SQL from production SQL.

📚 The Book Index Analogy

A database index is like the index at the back of a textbook. Without it, finding "GROUP BY" means reading every single page. With it, you jump directly to page 142. An index trades storage space for lookup speed. Add them on columns you frequently filter or join on.

Subquery Patterns

SQL — Subqueries
-- Scalar subquery: single value as a filter threshold
SELECT name, total_amount
FROM orders
WHERE total_amount > (SELECT AVG(total_amount) FROM orders);

-- IN subquery: filter against a result set
SELECT name, city
FROM customers
WHERE customer_id IN (
    SELECT customer_id FROM orders
    WHERE total_amount > 500
);

-- EXISTS: efficient alternative to IN for large sets
SELECT c.name
FROM customers c
WHERE EXISTS (
    SELECT 1 FROM orders o
    WHERE o.customer_id = c.customer_id
    AND o.total_amount > 1000
);

Creating Indexes

SQL — Indexes
-- Single-column index: speed up WHERE city = '...'
CREATE INDEX idx_customers_city ON customers(city);

-- Composite index: speed up WHERE city='...' AND signup_date > '...'
CREATE INDEX idx_customers_city_date ON customers(city, signup_date);

-- Always index foreign keys — they're used in JOINs constantly
CREATE INDEX idx_orders_customer ON orders(customer_id);
⚠️ Index Trade-offs

Indexes speed up reads but slow down writes (INSERT/UPDATE/DELETE must update the index too). Don't index every column. Index the columns that appear most in WHERE clauses and JOIN conditions. A table with 20 indexes writes 20× slower.

Phase 3 · Python Lesson 12

NumPy — The Speed Engine

A pure Python loop over 1 million numbers takes ~100ms. The same operation in NumPy takes ~1ms. This isn't a small improvement. It changes what's possible.

🍕 The Food Processor Analogy

Python loops are a chef chopping vegetables one by one. NumPy is a food processor — the entire batch, simultaneously, in C. Vectorization means "apply to all at once."

Python — NumPy
import numpy as np
arr = np.array([1, 2, 3, 4, 5])
print(arr * 2)         # [2, 4, 6, 8, 10]
print(arr[arr > 3])    # boolean indexing: [4, 5]
a = np.array([[1],[2],[3]])
b = np.array([10,20,30])
print(a + b)  # broadcasting — shape (3,3)

Phase 4 · Data Science Lesson 16

Statistical Thinking

You can run any model without understanding statistics. You can't interpret it without statistics. The models are the easy part. Understanding what the numbers mean is the hard part. This is where good data scientists separate from the rest.

🎯 Learning Objectives · ⏱ 25 min

After this lesson you'll be able to: Choose mean vs median for a given dataset · Explain what a p-value actually means (without the jargon) · Identify when correlation is being confused with causation · Construct a confidence interval interpretation.

📊
Distribution
How values spread. Normal, skewed, bimodal. Every dataset has a shape.
🎯
Central Tendency
Mean vs median. Mean is pulled by outliers. Median isn't. Use median for income.
📏
Variance & Std Dev
How spread out the data is. High variance = unpredictable. Low variance = consistent.
🔗
Correlation
How two variables move together. -1 to +1. Correlation ≠ causation.
🎲
p-value
"Assuming nothing is happening, how likely is this result by chance?" p < 0.05 = significant.
📦
Confidence Interval
A range where the true value probably lies. "95% CI: [48, 52]" is more honest.
Phase 4 · Data Science Lesson 17

EDA — Exploratory Data Analysis

EDA is the most underrated skill in data science. The best practitioners spend 60% of their time here, before touching a model. The model is the last 10%.

🎯 Learning Objectives · ⏱ 30 min

After this lesson: Execute a full EDA in under 10 lines of code · Detect null patterns, outliers, and skew visually · Build a correlation heatmap and interpret it · Document 5 business insights from a raw dataset.

🗺️ The EDA Framework

1. Shape – rows, columns, dtypes. 2. Nulls – where and why. 3. Distributions – each column's shape. 4. Relationships – how columns relate. 5. Anomalies – what's weird or interesting.

Python — EDA Template
import pandas as pd, matplotlib.pyplot as plt, seaborn as sns
df = pd.read_csv('data.csv')
print(f"Shape: {df.shape}")
print(df.dtypes)
null_pct = (df.isnull().sum() / len(df) * 100).sort_values(ascending=False)
print(null_pct[null_pct > 0])
num_cols = df.select_dtypes(include='number').columns
df[num_cols].hist(bins=30, figsize=(12,8))
sns.heatmap(df[num_cols].corr(), annot=True, cmap='coolwarm')
sns.boxplot(data=df[num_cols])
📂 Practice Dataset

Use the Superstore dataset — the best beginner EDA dataset: kaggle.com → Superstore Sales. It has sales, categories, regions, and dates — perfect for practicing every EDA step above.

⚠️ Common EDA Mistakes

Don't: Skip checking data types — a numeric column stored as text will give wrong results silently. Don't: Delete outliers immediately — first understand why they exist (data error vs. genuine extreme value). Don't: Confuse correlation in a heatmap with causation. Do: Always check your df.shape before and after any filtering.

Phase 4 · Data Science Lesson 18

Feature Engineering

Raw data is almost never in the right shape for a model. Feature engineering is the craft of transforming raw signals into information the model can actually use. This is where domain knowledge meets data science.

🏆 The Secret of Kaggle Winners

Top Kaggle competitors don't win with fancier models. They win with better features. A great feature on a simple model beats a mediocre feature on a complex model, every time.

Mini Case: Predicting Customer Churn

You have a raw orders table. The model can't use dates or IDs directly — you need to engineer signals it can learn from:

Raw ColumnEngineered FeatureWhy it helps
signup_datetenure_days = today − signup_dateLonger customers churn less — tenure is predictive
order_datesdays_since_last_orderRecency is the strongest churn signal
total_amountavg_order_value = total / order_countHigh-value customers behave differently
region (text)One-hot encoded columnsModels need numbers, not strings
revenue (skewed)log_revenue = log(1 + revenue)Compresses outliers; helps linear models
Python — Churn Feature Engineering
import pandas as pd, numpy as np
# Temporal features
df['tenure_days'] = (pd.Timestamp('today') - df['signup_date']).dt.days
df['days_since_last_order'] = (pd.Timestamp('today') - df['last_order_date']).dt.days
df['month'] = df['order_date'].dt.month
df['is_weekend'] = df['order_date'].dt.dayofweek >= 5
# Value features
df['avg_order_value'] = df['total_revenue'] / df['order_count'].clip(lower=1)
df['log_revenue'] = np.log1p(df['total_revenue'])
df['rolling_7d_avg'] = df['revenue'].rolling(7).mean()
# Categorical encoding
df = pd.get_dummies(df, columns=['region', 'category'], drop_first=True)
⚠️ Common Feature Engineering Mistakes

Data leakage: Never use features that include information from the future (e.g., using next month's orders to predict this month's churn). Scaling mismatch: Scale features after train/test split — never before, or your test set leaks into training. Too many features: Adding weak features adds noise. Less than 20 strong features usually beats 200 weak ones.

Phase 4 · Data Science Lesson 19

Model Evaluation & Tuning

Accuracy is a lie. 95% accuracy on a fraud dataset where 95% of transactions are legitimate means your model learned to say "not fraud" every time. Evaluation is the science of knowing if your model actually works.

MetricWhen to use itWhat it means
AccuracyBalanced classes only% correct overall
PrecisionFalse positives are costly"Of predicted positives, how many were right?"
RecallFalse negatives are costly"Of actual positives, how many did I catch?"
F1 ScoreImbalanced classesHarmonic mean of precision + recall
RMSERegression problemsAverage prediction error in original units
R²Regression problems"How much variance do I explain?" 1.0 is perfect

param_grid = {'model__n_estimators':[50,100,200], 'model__max_depth':[None,5,10]}

Streamlit Boilerplate — Your First App

Install with pip install streamlit pandas matplotlib, paste this, then run streamlit run app.py:

Python — Streamlit Dashboard Starter
import streamlit as st
import pandas as pd
import matplotlib.pyplot as plt

st.set_page_config(page_title="Sales Dashboard", layout="wide")
st.title("📊 Sales Dashboard")

# 1. Load data (replace with your CSV path)
df = pd.read_csv("superstore.csv", parse_dates=["Order Date"])

# 2. Sidebar filters
regions = st.sidebar.multiselect("Region", df["Region"].unique(), default=df["Region"].unique())
df = df[df["Region"].isin(regions)]

# 3. KPI cards
col1, col2, col3 = st.columns(3)
col1.metric("Total Revenue", f"${df['Sales'].sum():,.0f}")
col2.metric("Total Orders", len(df))
col3.metric("Avg Order Value", f"${df['Sales'].mean():,.2f}")

# 4. Chart
monthly = df.groupby(df["Order Date"].dt.to_period("M"))["Sales"].sum()
st.line_chart(monthly)

# 5. Data table
st.dataframe(df.head(20))

Phase 5 · Build Lesson 21

Dashboard + Storytelling

The best data scientist who can't communicate their findings is less valuable than a mediocre analyst who can. The last skill is translating numbers into decisions.

📰 The Journalist Rule

A good journalist leads with the most important finding, not the chronological story. Your data presentation should work the same way: Insight first. Evidence second. Detail last. Don't make your audience hunt for the point.

Dashboard Design Principles

  • One dashboard, one question. "How is the business doing this month?" not "Here is everything about everything."
  • The 5-second rule. The most important insight should be readable in 5 seconds without reading a title.
  • Color has meaning. Red = bad, green = good. Saturation = intensity. Don't use color just to look nice.
  • Remove what doesn't earn its place. Every gridline, legend entry, and label should justify its presence.
